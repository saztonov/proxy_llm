# Подключение портала к LLM-прокси `proxy_llm`

`proxy_llm` — центральный прокси между вашим сервисом и OpenRouter. Он прячет реальный
ключ OpenRouter, ведёт журнал вызовов, шлёт алерты и применяет пер-клиентскую политику
моделей и лимитов. Для клиента это **OpenAI-совместимый Chat Completions API**: любой
OpenAI-клиент подключается сменой `base_url` и `api_key`, без переписывания логики.

**Скилл самодостаточный и переносимый.** Скопируйте каталог `connect-proxy-llm/` в
`.claude/skills/` своего проекта. Здесь есть всё для интеграции: адрес, параметры всех
корпоративных порталов, оба режима работы с моделью, готовые клиенты, коды ошибок и
приёмочные проверки. Операторских внутренностей нет.

**Единственное, чего здесь нет и не будет, — токен.** Его выдаёт оператор отдельным
защищённым каналом.

Навигация:
- **Подключаетесь впервые** → §0, затем выберите вариант в §1.
- **Уже подключены на дефолтной модели, хотите выбирать сами** → §9.
- *(оператору прокси: онбординг клиента и включение выбора модели — `docs/vps-update.md`
  §4 и §4a, проверка дедупа — `docs/operator-guide.md`. В этом скилле их нет намеренно,
  чтобы его можно было отдать порталу как есть.)*

## 0. Параметры подключения

**Базовый URL (прод): `https://proxyllm.fvds.ru`**

Общее для всех порталов:

| Параметр | Значение |
|---|---|
| Эндпоинт | `POST https://proxyllm.fvds.ru/api/v1/chat/completions` |
| Liveness (без токена) | `GET https://proxyllm.fvds.ru/healthz` |
| Дефолтная модель (при заглушке `proxy`) | `google/gemini-3.1-flash-lite` |
| Fallback-цепочка (при заглушке) | `google/gemini-3-flash-preview` |
| Дедлайн запроса | 190 с |
| Лимит тела запроса | 26 МБ |
| Общая очередь прокси | 3 одновременных, 6 в ожидании |

Параметры порталов:

| `clientId` | Вариант (§1) | Может выбирать модель | Лимиты (одновременно / очередь) |
|---|---|---|---|
| `passdesk` | **A** | нет, `model` игнорируется | 2 / 3 |
| `fot` | **B** — `allowedModels: ["*"]` | **да, любую модель OpenRouter** | 1 / 2 |
| `matcheck` | **A** | нет, `model` игнорируется | 1 / 2 |
| `estimat` | **A** | нет, `model` игнорируется | 1 / 2 |

Ключ OpenRouter общий на все порталы — персональных нет, расходы идут по общему бюджету.
Нужен изолированный биллинг — запросить у оператора `openrouterApiKey`.

**Единственное, чего здесь нет, — токен.** Он передаётся отдельным защищённым каналом и
не должен попадать ни в этот файл, ни в репозиторий, ни в код. Токен идентифицирует ваш
`clientId`: к нему привязаны лимиты, политика моделей и журнал. Не переиспользовать между
проектами и не передавать соседним порталам. Хранить в секрете сервиса (env/secret-storage).

egress-IP всех четырёх порталов уже в nginx-allowlist. Новый портал или смена адреса —
запросить у оператора **до** начала работ: иначе прокси откажет ещё до проверки токена.

Клиентские env-переменные (имена — на ваш вкус, здесь по всему скиллу используются эти):

```env
PROXY_LLM_BASE_URL=https://proxyllm.fvds.ru
PROXY_LLM_TOKEN=<из защищённого канала>
PROXY_LLM_MODEL=proxy
```

**Заводите `PROXY_LLM_MODEL` с самого начала, даже в варианте A.** Значение `proxy` —
заглушка «модель выбирает прокси». Тогда переход A → B (§9) будет сменой значения env, а не
правкой кода, и откат тоже.

## 1. Два варианта работы с моделью

Политику задаёт **оператор** пер-клиентски, полем `allowedModels` в реестре. Вы её не
меняете — но обязаны знать свою, потому что от неё зависит поведение поля `model`.

### Вариант A — модель выбирает прокси (`allowedModels` пусто)

Поведение по умолчанию для нового клиента. Присланный `model` **игнорируется** —
всегда идёт дефолтная модель вашего клиента плюс её fallback-цепочка.

- **Кому:** порталу, которому нужен «просто LLM», без требований к конкретной модели.
- **Плюс:** отказоустойчивость. Недоступен провайдер — прокси сам переезжает на резервную
  модель из `fallbackModels`, вы этого даже не замечаете.
- **Плюс:** оператор может сменить модель всем сразу, без релиза на вашей стороне.
- **Минус:** вы не управляете качеством и ценой вызова.
- **В коде:** шлите заглушку `proxy` либо не шлите `model` вообще.
- **`400 model_not_allowed` не бывает никогда.**

### Вариант B — модель выбирает портал, любая из OpenRouter (`allowedModels: ["*"]`)

Можно прислать **любой** слаг из каталога OpenRouter (`https://openrouter.ai/models`).

- **Кому:** порталу, которому нужна конкретная модель — из-за качества, длины контекста,
  мультимодальности или цены.
- **Плюс:** полный контроль, можно выбирать модель на каждый вызов отдельно.
- **Минус — отказоустойчивость:** явный выбор **отключает fallback-цепочку** (см. ниже).
- **Минус — нет страховки от опечатки:** прокси **не проверяет существование модели**.
  Неверный слаг уходит в OpenRouter и возвращается **его** ошибкой, а не `model_not_allowed`:

  ```json
  {"message":"no/such-model is not a valid model ID","code":400}
  ```

  Поэтому **логируйте значение `PROXY_LLM_MODEL` в тексте ошибки** — иначе диагностика
  опечатки в env займёт часы.
- **Минус — биллинг:** модели отличаются по цене в разы. Смена ручки меняет счёт
  немедленно и на весь трафик портала. Сверяйтесь с ценой на `https://openrouter.ai/models`
  до переключения.
- **`400 model_not_allowed` не бывает никогда** (проверять нечему).

### Вариант C — белый список (`allowedModels: ["a/b","c/d"]`)

Промежуточный: выбор разрешён, но только из перечисленных оператором слагов.

- **Кому:** когда нужен выбор, но с потолком по расходам или фиксированным набором.
- **Единственный вариант, где бывает `400 model_not_allowed`** — модель вне списка:

  ```json
  {"error":{"code":"model_not_allowed","message":"requested model is not allowed for this client","allowed":["google/gemini-3.1-flash-lite","openai/gpt-5.6-luna-pro"]}}
  ```

  Не ретраить — это конфиг, а не сбой. Прочитать поле `allowed`, залогировать.
- Глобов нет: `["anthropic/*"]` не работает, только точные слаги либо ровно `["*"]`.

### Сводка

| | A — дефолт | B — любая | C — список |
|---|---|---|---|
| `allowedModels` у оператора | пусто | `["*"]` | `["a/b", …]` |
| что слать в `model` | `proxy` / ничего | любой слаг | слаг из списка |
| fallback-цепочка | **работает** | отключается при выборе | отключается при выборе |
| `400 model_not_allowed` | никогда | никогда | **бывает** |
| плохой слаг | игнорируется | ошибка OpenRouter | `400 model_not_allowed` |
| контроль расходов | у оператора | **у вас** | у оператора (потолок) |

### Заглушки — работают во всех трёх вариантах

`proxy`, `default`, `auto` в поле `model` означают «модель не выбираю» → дефолтная модель
клиента **плюс fallback**. Так же трактуются отсутствующий и пустой `model`. Регистр и
пробелы по краям не важны.

Это **единственный** безопасный способ прислать «пустышку». OpenAI SDK требует непустой
`model`, и заглушки существуют именно ради него. Любая другая строка в вариантах B и C —
это **реальный выбор**, который изменит роутинг и биллинг.

Сравнение по всей строке: реальный слаг `openrouter/auto` заглушкой **не** считается.

### Явный выбор отключает fallback

Прислали конкретную модель → в OpenRouter уходит одиночный `model` без `models[]`:
недоступен провайдер — получите ошибку, а не тихий переезд на резервную модель.

Нужна отказоустойчивость при варианте B/C — либо не выбирайте модель (заглушка), либо
реализуйте перебор моделей у себя: поймали ошибку — повторили с другим слагом.

### Как убедиться, какая модель реально отработала

**Не спрашивайте об этом саму модель — её самоидентификация не доказательство.** Проверено:

| запрошено | что модель отвечает про себя |
|---|---|
| `openai/gpt-5.6-luna-pro` | «Я — ChatGPT, модель **GPT-5** от OpenAI» |
| `google/gemini-3.1-flash-lite` | «Я — большая языковая модель, обученная Google» |

Точную версию по такому ответу не определить. Достоверны только метаданные ответа:

- **поле `model` в теле ответа** — его проставляет OpenRouter по факту исполнения;
- **заголовок `x-openrouter-request-id`** (`gen-…`) — по нему оператор находит конкретный
  вызов в биллинге OpenRouter и подтверждает модель независимо от вас. Логируйте его на
  каждый вызов: это единственный способ доказать что-либо задним числом;
- **`usage.completion_tokens_details.reasoning_tokens`** — ненулевое значение подтверждает,
  что отработала reasoning-модель.

В вариантах B и C при явном выборе fallback отключён, поэтому подмены модели «за спиной»
быть не может: либо отработает запрошенная, либо придёт ошибка. Расхождение `model` с
запрошенным слагом — повод идти к оператору, **кроме** случая из следующего раздела.

### Управление reasoning

Поля `reasoning`, `reasoning_effort` и `include_reasoning` не вырезаются прокси и доходят
до модели как есть. Поддержку смотрите в `supported_parameters` модели на
`https://openrouter.ai/models`.

Допустимые `effort`: `max`, `xhigh`, `high`, `medium`, `low`, `minimal`, `none`.
Невалидное значение → 400 от OpenRouter (не в конверте прокси), не ретраить.

Три вещи, на которых спотыкаются:

1. **`reasoning.mode` может изменить слаг в ответе.** У некоторых семейств «pro»-вариант —
   это та же модель с другим режимом рассуждений. Запрос `openai/gpt-5.6-luna` с
   `"reasoning":{"mode":"pro"}` возвращает `model: openai/gpt-5.6-luna-pro`. Поэтому
   **не пишите жёсткую проверку `response.model === request.model`.**
2. **Текст рассуждений отдают не все.** У моделей OpenAI `message.reasoning` приходит
   `null` даже при `include_reasoning: true` — виден только счётчик `reasoning_tokens`.
   Gemini, наоборот, возвращает полный подписанный `reasoning_details`. Если ответ
   показывается конечному пользователю, а рассуждения показывать нельзя — либо берите
   модель, которая их не отдаёт, либо вырезайте поле сами.
3. **Reasoning съедает бюджет `max_tokens` раньше контента.** При нехватке приходит
   **200 OK с `content: null`** и `finish_reason: "length"` — и вызов всё равно оплачен.
   Это выглядит как молчаливая порча данных, а не как ошибка. Проверяйте всегда:

   ```js
   if (choice.finish_reason === 'length' || !choice.message.content) {
     // не успех: перезапросить с большим max_tokens, не отдавать пользователю
   }
   ```

Reasoning-токены тарифицируются как выходные и учитываются в дедлайне 190 с: высокий
`effort` на большом промпте — реальный риск `504 deadline_exceeded`.

## 2. Ограничения прокси

Прочитайте до интеграции — прокси намеренно отличается от прямого OpenAI/OpenRouter:

- **Один эндпоинт:** `POST /api/v1/chat/completions` (легаси-алиас `POST /v1/chat/completions`,
  идентичен). Эндпоинта `/v1/models` **нет**, Anthropic `/v1/messages` **нет**. Список
  доступных моделей программно не узнать: в варианте B сверяйтесь с каталогом OpenRouter,
  в варианте C — со списком от оператора (он же приходит в поле `allowed` при 400).
- **Стриминг запрещён.** `"stream": true` → **HTTP 400** `streaming_not_supported`.
  Только non-streaming запрос/ответ.
- **Молча вырезаются** из тела: `model` (переписывается по политике), `models`, `provider`,
  `route`, `transforms`, `plugins`, `stream`, `stream_options`, `debug` — защита от обхода
  серверного роутинга и провайдера. В частности, **свою fallback-цепочку `models[]` прислать
  нельзя**, её задаёт оператор в `fallbackModels`.
- **Дедлайн запроса** ~190 с (`REQUEST_DEADLINE_MS`) → при исчерпании **504** `deadline_exceeded`.
- **Лимит тела запроса** ~26 МБ → **413** `payload_too_large`.
- **Два потолка конкурентности:** ваш пер-клиентский (`maxConcurrency`/`maxPending`, уточните
  у оператора) и общий на весь прокси. Переполнение любого → **503** `queue_full` +
  `Retry-After: 10`. Обрабатывайте ретраем с backoff (см. §7).
- **Только server-to-server.** CORS не настроен — **из браузера не вызывать**.
- **Ответ** — стандартный OpenAI JSON (`id`, `model`, `choices`, `usage`), плюс заголовки
  `x-proxy-request-id` (id запроса в прокси) и `x-openrouter-request-id` (upstream `gen-…`
  для биллинг-сверки).
- **Порядок проверок:** IP-allowlist → токен → тело. Не тот IP — до токена дело не дойдёт.

## 3. Аутентификация

Каждый запрос предъявляет ваш персональный токен в заголовке:

```
Authorization: Bearer <PROXY_LLM_TOKEN>
```

Токен идентифицирует ваш `clientId` — к нему привязаны лимиты, политика моделей и журнал.
Отсутствует/неверный → **401** `unauthorized`. Проверка идёт до чтения тела.

## 4. Быстрые проверки (curl)

```bash
BASE=https://proxyllm.fvds.ru
TOKEN=<ваш-токен>

# Публичный liveness (без токена)
curl -s "$BASE/healthz"                    # -> {"status":"ok"}

# Минимальный вызов: model не шлём — идёт дефолтная модель вашего клиента.
curl -s "$BASE/api/v1/chat/completions" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -H "X-Request-Id: req-$(date +%s)-$RANDOM" \
  -d '{"messages":[{"role":"user","content":"ping"}]}'
# -> OpenAI-ответ: {"id":...,"model":...,"choices":[...],"usage":{...}}
#    заголовки ответа: x-proxy-request-id, x-openrouter-request-id

# Заглушка — эквивалент отсутствия model. Работает во всех вариантах:
curl -s "$BASE/api/v1/chat/completions" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"model":"proxy","messages":[{"role":"user","content":"ping"}]}'
# -> 200, в ответе model = дефолтная модель вашего клиента

# Явный выбор (варианты B/C; ОТКЛЮЧАЕТ fallback-цепочку):
curl -s "$BASE/api/v1/chat/completions" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"model":"openai/gpt-5.6-luna-pro","messages":[{"role":"user","content":"ping"}]}'
# -> 200, в ответе model = запрошенная

# Заведомо неверный слаг — реакция ЗАВИСИТ ОТ ВАШЕГО ВАРИАНТА:
curl -s "$BASE/api/v1/chat/completions" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"model":"no/such-model","messages":[{"role":"user","content":"x"}]}'
# A -> 200, слаг проигнорирован, отработала дефолтная модель
# B -> 400 {"message":"no/such-model is not a valid model ID","code":400}   (ошибка OpenRouter)
# C -> 400 {"error":{"code":"model_not_allowed","allowed":[...]}}           (конверт прокси)

# Стриминг запрещён:
curl -s "$BASE/api/v1/chat/completions" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"messages":[{"role":"user","content":"hi"}],"stream":true}'
# -> 400 {"error":{"code":"streaming_not_supported",...}}

# Без токена:
curl -s -o /dev/null -w '%{http_code}\n' "$BASE/api/v1/chat/completions" \
  -H "Content-Type: application/json" -d '{"messages":[{"role":"user","content":"x"}]}'
# -> 401
```

Проверка «неверный слаг» — самый быстрый способ **узнать свой вариант на практике**, если
ответ оператора потерялся.

> `uuidgen` в примерах намеренно не используется: его нет по умолчанию в Windows/Git Bash.
> В коде берите штатный UUID языка (`crypto.randomUUID()`, `uuid.uuid4()`).

## 5. Заголовки трассировки и дедупа

| Заголовок | Обяз. | Назначение | Ограничения |
|---|---|---|---|
| `X-Idempotency-Key` | **да, если у вас есть ретраи** | Стабильный ключ **логической задачи**. Конкурентные запросы с одинаковым ключом схлопываются в **один** upstream-вызов (дедуп). | ≤256 символов |
| `X-Request-Id` | рекомендуется | Трассировка одной **попытки**. Уникальный на каждую попытку (UUID). | ≤128 символов, иначе прокси сгенерит свой |

### 5.1 `X-Idempotency-Key` — обязателен при ретраях

Если что-то в вашем сервисе повторяет упавший LLM-вызов — очередь (BullMQ, Celery, Sidekiq),
cron, ретрай в HTTP-клиенте или живой человек, жмущий «повторить», — ключ **обязателен**.
Без него дедуп прокси не включается **никогда**, и каждый ретрай уходит в OpenRouter
**новым платным вызовом** вместо схлопывания с исходным. На запросах со сканами и картинками
это сотни килобайт и полный счёт за токены на каждую повторную попытку. Прокси починить это
за вас не может: связать попытки без ключа физически нечем.

Ключ можно не слать, только если ретраев нет вообще нигде.

**Слать из ВСЕХ мест, где вы зовёте прокси.** Это главная ошибка на практике: в сервисе
обычно не один `axios.post`/`requests.post`, а несколько — основной путь, батч, другой тип
документа, легаси-ветка. Заголовок добавляют в одно место, интеграцию считают сделанной, а
боевой трафик годами идёт мимо дедупа. Найдите все места вызова и проверьте каждое:

```bash
# подставьте свой способ вызова и свои имена env-переменных
grep -rn "chat/completions\|PROXY_LLM_BASE_URL" src/ server/ --include='*.js' --include='*.ts' --include='*.py'
```

Если проект раньше ходил в OpenRouter напрямую, добавьте в grep **старые** имена
переменных — легаси-ветка обычно живёт именно под ними.

Правило значения ключа:
- **должен совпадать** между ретраями одной и той же задачи;
- **не должен совпадать** между разными задачами/входами/промптами;
- при смене промпта/схемы поднимайте версию в ключе, чтобы старые ключи не конфликтовали.

Ключ скоупится по вашему `clientId`, поэтому совпадение ключа с другим клиентом безопасно.

Стабильный ключ обычно строят как хэш от устойчивых атрибутов задачи (пример — sha256):

```js
const crypto = require('node:crypto');
const idempVer = process.env.PROXY_LLM_IDEMPOTENCY_VERSION || 'v1';
const input = [taskId, inputHashOrKey, promptNameOrHash, idempVer].filter(Boolean).join(':');
const idempotencyKey = crypto.createHash('sha256').update(input).digest('hex');
```

**Если вы выбираете модель (B/C) — включайте слаг в ключ.** Иначе повтор той же задачи на
другой модели схлопнется с прежним вызовом и вернёт ответ старой модели.

В ответе смотрите `x-proxy-request-id` / `x-openrouter-request-id` для логов и сверки
биллинга (по `x-openrouter-request-id = gen-…` вызов находится в биллинге OpenRouter).

## 6. Готовые клиенты

Все примеры читают `PROXY_LLM_*` из окружения — токен в коде не хардкодить. Вариант A или B
задаётся **только значением `PROXY_LLM_MODEL`**, код одинаковый:

```env
PROXY_LLM_MODEL=proxy                       # вариант A — модель выбирает прокси
PROXY_LLM_MODEL=openai/gpt-5.6-luna-pro     # вариант B/C — выбираем сами
```

### 6.1 Python — OpenAI SDK

`base_url` указывает на `…/api/v1`, SDK сам добавит `/chat/completions`.

```python
import os
from openai import OpenAI

client = OpenAI(
    base_url=f"{os.environ['PROXY_LLM_BASE_URL']}/api/v1",
    api_key=os.environ["PROXY_LLM_TOKEN"],
)

resp = client.chat.completions.create(
    # SDK требует непустой model. "proxy" = заглушка «модель не выбираю» -> дефолт клиента
    # + fallback. Реальный слаг = явный выбор (нужен вариант B/C, fallback отключится).
    model=os.environ.get("PROXY_LLM_MODEL", "proxy"),
    messages=[{"role": "user", "content": "ping"}],
    # stream НЕ задавать: stream=True -> 400
    extra_headers={"X-Request-Id": os.urandom(16).hex()},
)
print(resp.choices[0].message.content)
```

### 6.2 Python — raw `requests` (полный контроль над кодами/ретраем)

```python
import os, uuid, requests

BASE  = os.environ["PROXY_LLM_BASE_URL"]
TOKEN = os.environ["PROXY_LLM_TOKEN"]
MODEL = os.environ.get("PROXY_LLM_MODEL", "proxy")   # "proxy" = заглушка

def chat(messages, idempotency_key=None):
    headers = {
        "Authorization": f"Bearer {TOKEN}",
        "Content-Type": "application/json",
        "X-Request-Id": str(uuid.uuid4()),          # новый на каждую попытку
    }
    if idempotency_key:
        headers["X-Idempotency-Key"] = idempotency_key  # стабильный на задачу
    r = requests.post(f"{BASE}/api/v1/chat/completions",
                      headers=headers,
                      json={"model": MODEL, "messages": messages},
                      timeout=200)
    if r.status_code >= 400:
        body = r.json() if "application/json" in r.headers.get("content-type", "") else {}
        err = body.get("error") or {}
        # Ошибка прокси: {"error":{"code":...}}. Ошибка OpenRouter — своя форма, без конверта.
        if err.get("code") == "model_not_allowed":   # только вариант C; не ретраить
            raise RuntimeError(f"model {MODEL!r} запрещена, разрешены: {err.get('allowed')}")
        # MODEL в тексте: в варианте B неверный слаг виден только так.
        raise RuntimeError(f"{r.status_code} {err.get('code', '')} (model={MODEL!r}): {r.text[:300]}")
    return r.json()
```

### 6.4 Node/TypeScript — мини-клиент на `fetch` (различает 503/504)

Модель принимается параметром на вызов — так вариант B позволяет выбирать её точечно, а не
глобально на весь портал.

```ts
import crypto from 'node:crypto';

class ProxyLLM {
  constructor(private base = process.env.PROXY_LLM_BASE_URL!,
              private token = process.env.PROXY_LLM_TOKEN!,
              private defaultModel = process.env.PROXY_LLM_MODEL || 'proxy') {}

  // { status, body } — чтобы вызывающий код различал 503 queue_full / 504 от прочего.
  async chat(messages: unknown[], opts: { idempotencyKey?: string; model?: string } = {}
  ): Promise<{ status: number; body: any }> {
    const headers: Record<string, string> = {
      Authorization: `Bearer ${this.token}`,
      'Content-Type': 'application/json',
      'X-Request-Id': crypto.randomUUID(),            // новый на каждую попытку
    };
    if (opts.idempotencyKey) headers['X-Idempotency-Key'] = opts.idempotencyKey; // стабильный на задачу
    const res = await fetch(`${this.base}/api/v1/chat/completions`, {
      method: 'POST',
      headers,
      body: JSON.stringify({ model: opts.model ?? this.defaultModel, messages }),
    });
    const body = await res.json().catch(() => ({}));
    return { status: res.status, body };
  }
}
```

## 7. Ошибки и ретраи

Тело ошибки, сгенерированной **прокси**: `{ "error": { "code": "...", "message": "..." } }`.

| код | HTTP | причина | что делать |
|---|---|---|---|
| `unauthorized` | 401 | нет/неверный Bearer | проверить токен |
| `invalid_request` | 400 | тело не JSON-объект или `messages` пустой/не массив | исправить тело |
| `streaming_not_supported` | 400 | прислан `stream:true` | убрать `stream` |
| `model_not_allowed` | 400 | `model` вне белого списка (`allowed` в теле). **Только вариант C** | взять модель из `allowed`, либо прислать заглушку `proxy`. Не ретраить |
| `payload_too_large` | 413 | тело > ~26 МБ | уменьшить запрос |
| `queue_full` | 503 | очередь прокси переполнена (есть `Retry-After`) | **ретрай** с backoff, уважая `Retry-After` |
| `dedup_full` | 503 | исчерпан лимит активных idempotency-ключей (есть `Retry-After`) | **ретрай** с backoff |
| `deadline_exceeded` | 504 | превышен серверный дедлайн ~190 с | **ретрай** (тот же idempotency-key) |
| `attempt_timeout` / `network_error` | 504 | таймаут/сбой связи прокси↔OpenRouter | **ретрай** |
| `upstream_response_too_large` | 502 | ответ OpenRouter превысил лимит | не ретраить, уменьшить задачу |
| `internal` | 500 | внутренняя ошибка прокси | ограниченный ретрай |

**Важно:** «настоящие» ошибки OpenRouter (401/402 по ключу OpenRouter, 429 rate limit, 5xx)
прокси **пробрасывает как есть** — со статусом и телом OpenRouter, а не в своём envelope.
То есть не всякий 4xx/5xx имеет форму `{error:{code}}`; ориентируйтесь на HTTP-статус в
первую очередь, а `error.code` читайте опционально.

**В варианте B это основной способ узнать о проблеме с моделью:** несуществующий,
снятый с публикации или недоступный слаг придёт ошибкой OpenRouter. Логируйте вместе с
ней значение `PROXY_LLM_MODEL`.

Правило ретрая на клиенте:
- ретраить **503 / 504 / сетевые** ошибки — экспоненциальный backoff, уважать `Retry-After`;
- при ретрае слать **тот же** `X-Idempotency-Key` и **новый** `X-Request-Id`;
- **не** ретраить 400/401/413 (кроме 429 — там backoff);
- прокси сам делает до 2 upstream-попыток внутри дедлайна, поэтому клиентский ретрай — это
  «внешний» слой на случай `queue_full`/дедлайна.

## 8. Мультитенантность: что изолировано, а что общее

Токен = ваш `clientId`. Прокси мультитенантный.

**Изолировано по клиентам:**
- токен — свой у каждого проекта, его ротация не задевает соседей;
- лимиты конкурентности `maxConcurrency`/`maxPending` — своя очередь;
- политика моделей: `defaultModel`, `allowedModels`, `fallbackModels`;
- журнал и статистика (`client_id`); серия ошибок для алертов считается пер-клиентски;
- дедуп по `X-Idempotency-Key`;
- опционально — свой ключ OpenRouter (биллинговая изоляция).

**Общее для всех:**
- глобальный потолок одновременных upstream-вызовов — под нагрузкой соседи всё ещё могут
  вас притормозить вплоть до `queue_full`;
- серверный дедлайн ~190 с и лимит тела ~26 МБ;
- ключ OpenRouter — если оператор не выдал вам персональный (общий бюджет и биллинг);
  в варианте B это стоит уточнить отдельно: **выбор модели = выбор биллинга**;
- nginx IP-allowlist — общий на `location /api/`, не пер-клиентский: привязки
  «токен X только с IP Y» нет;
- дайджест в Telegram — агрегат по всему прокси, без разбивки.

**Чего нет:** rate limit и дневных квот на клиента (лимит одновременности — это
back-pressure, не throttling); самообслуживания — смена модели/лимитов/IP только через
оператора, реестр читается на старте сервиса.

## 9. Переход с варианта A на вариант B

Для портала, который **уже подключён** на дефолтной модели и хочет выбирать сам.

Доработка кода (шаги 2-5) и включение выбора оператором (шаг 6) независимы и безопасны в
любом порядке: пока `PROXY_LLM_MODEL=proxy`, доработанный клиент ведёт себя ровно как
недоработанный.

Раз уж вы всё равно открыли блок сборки `headers` — **сначала проверьте `X-Idempotency-Key`
во всех местах вызова (§5.1)**. Он правится там же, а стоит дороже: пропущенное место вызова
оплачивается двойными вызовами на каждом ретрае прямо сейчас, тогда как выбор модели — это
возможность на будущее. Порядок: сперва ключ, потом модель.

1. **Проверить, что вы шлёте сейчас.** Найдите в коде **все** места вызова (grep из §5.1) и
   посмотрите, что каждое кладёт в `model`. Три случая:
   - **реальный слаг** (например, унаследованный `*_MODEL` из прежней прямой интеграции с
     OpenRouter) — **осторожно**: сейчас прокси его игнорирует, но как только оператор
     включит вам выбор, значение **оживёт** и молча изменит роутинг и биллинг, а
     fallback-цепочка отключится. Сначала шаг 2, только потом просите включение;
   - **заглушка `"proxy"`** — безопасно, ничего не изменится;
   - **поле не шлётся** — безопасно.
2. **Завести ручку модели в конфиге**, а не хардкодить: `PROXY_LLM_MODEL=proxy`. Значение по
   умолчанию — заглушка, поэтому саму доработку можно выкатывать отдельно и незаметно.
3. **Слать значение ручки в `model`:** `model: process.env.PROXY_LLM_MODEL || 'proxy'`.
4. **Обработать ошибку модели** — какую именно, зависит от целевого варианта:
   B — ошибка OpenRouter (логировать вместе со значением ручки),
   C — `400 model_not_allowed` с полем `allowed`. Ни ту, ни другую не ретраить. Пример — §6.2.
5. **Учесть, что явный выбор отключает fallback-цепочку.** Прокси перестаёт подставлять
   `models[]`: недоступна модель — придёт ошибка, а не тихий переезд на резервную. Нужна
   отказоустойчивость — либо не выбирать модель, либо реализовать перебор моделей у себя.
6. **Попросить оператора** включить выбор для вашего `clientId`: `allowedModels: ["*"]`
   (вариант B) или конкретный список (вариант C), затем рестарт прокси. Заодно уточнить,
   идёте ли вы по общему ключу OpenRouter или по своему.
7. **Проверить** (§4): `PROXY_LLM_MODEL=proxy` → в ответе дефолтная модель;
   `PROXY_LLM_MODEL=<слаг>` → в ответе он же; заведомо неверный слаг → ошибка нужного вида.
8. **Сверить цену** нового слага на `https://openrouter.ai/models` до переключения ручки:
   переключение действует сразу и на весь трафик портала.

Откат — вернуть `PROXY_LLM_MODEL=proxy`: без правок кода и без участия оператора.

## 10. Чеклист интеграции

- [ ] Найден свой `clientId` в таблице §0 и понятен свой вариант (A / B / C).
- [ ] Токен получен от оператора по защищённому каналу; лежит в секретах сервиса,
      не в коде/репозитории; только HTTPS.
- [ ] `base_url` = `https://proxyllm.fvds.ru/api/v1` (для OpenAI SDK) либо POST на
      `https://proxyllm.fvds.ru/api/v1/chat/completions`.
- [ ] Модель в конфиге (`PROXY_LLM_MODEL`), а не в коде — даже если сейчас вариант A.
- [ ] **Вариант A:** шлём заглушку `proxy` либо не шлём `model`; произвольной
      строки-«пустышки» в `model` нет.
- [ ] **Вариант B:** слаг сверен с каталогом OpenRouter и по цене; ошибка OpenRouter
      обрабатывается и логируется вместе со значением `PROXY_LLM_MODEL`.
- [ ] **Вариант C:** слаг из списка оператора; обрабатывается 400 `model_not_allowed`.
- [ ] **B/C:** учтено, что явный выбор отключает fallback-цепочку; слаг входит в
      `X-Idempotency-Key`.
- [ ] `stream:true` не отправляется.
- [ ] Найдены **все** места вызова прокси (grep из §5.1), а не только основной путь.
- [ ] `X-Idempotency-Key` отправляется из **каждого** из них — если у сервиса есть ретраи.
      Проверено не по коду, а по факту: **оператор подтвердил**, что в журнале нет запросов
      с пустым ключом. Сами вы свой трафик в журнале не видите.
- [ ] Обрабатываются 503/504 с ретраем и backoff; при ретрае — тот же `X-Idempotency-Key`,
      новый `X-Request-Id`.
- [ ] Ориентация на HTTP-статус (не всякая ошибка имеет `{error:{code}}`).
- [ ] Вызовы только server-to-server (не из браузера).
- [ ] `x-openrouter-request-id` логируется на каждый вызов — без него модель и стоимость
      задним числом не подтвердить.
- [ ] Модель проверяется по полю `model` в ответе, а **не** вопросом к самой модели;
      жёсткого `response.model === request.model` нет (ломается на `reasoning.mode`).
- [ ] Если используется reasoning — `max_tokens` с запасом и проверка
      `finish_reason !== 'length'` перед выдачей результата пользователю.

## Проверка перед сдачей интеграции

Прогнать против `https://proxyllm.fvds.ru` **с сервера портала** — с рабочей станции не
выйдет, IP-allowlist пропускает только egress-адреса порталов.

1. `GET /healthz` → `200 {"status":"ok"}`.
2. `POST /api/v1/chat/completions` с корректным Bearer и `messages` → `200`, OpenAI-тело,
   в заголовках ответа есть `x-proxy-request-id`.
3. Тот же запрос без/с неверным токеном → `401 unauthorized`.
4. Тело с `"stream": true` → `400 streaming_not_supported`.
5. `{"model":"proxy", …}` → `200`, в теле ответа `model` = дефолтная модель клиента
   (заглушка не ушла в роутинг). **Проверять во всех вариантах** — это гарантия отката.
6. **B/C:** нужный слаг → `200`, в ответе `model` = запрошенная.
7. Заведомо неверный слаг → реакция, соответствующая вашему варианту (таблица §1).
   Расхождение означает, что ваш вариант на прокси не тот, о котором вы договорились.
8. Примеры Python/Node запускаются после подстановки `PROXY_LLM_BASE_URL` / `PROXY_LLM_TOKEN`.

**Дедуп — проверять фактом, а не глазами по коду.** Два параллельных запроса с одинаковым
`X-Idempotency-Key` должны дать **один** вызов в OpenRouter:

```bash
KEY="selftest-$(date +%s)"
for i in 1 2; do
  curl -s -o /dev/null -w "%{http_code} " "$BASE/api/v1/chat/completions" \
    -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
    -H "X-Idempotency-Key: $KEY" -H "X-Request-Id: req-$i-$RANDOM" \
    -d '{"model":"proxy","messages":[{"role":"user","content":"dedup selftest"}]}' &
done; wait; echo
# -> два 200. У оператора в журнале при этом ОДИН upstream-вызов на два request_id.
```

Главная проверка — **боевая, через сутки после запуска**: попросите оператора показать долю
ваших запросов без ключа. Она обязана быть нулевой. Запрос выполняет оператор — доступа к
журналу у вас нет.

Ненулевая доля означает, что какое-то место вызова осталось недоработанным — и именно оно,
скорее всего, несёт основной трафик.
